# 女仆仓管扩展

面向 Minecraft 1.20.1 / Forge 47.x 的非官方社区扩展，正在从仓管辅助逐步演进为**女仆物流网络**：以仓库和物资流转为基础，联动仓管、快递员、司机等不同职位的女仆，让物资调度、跨区域运输与人员接送组成一套可持续拓展的物流体系。

> [!IMPORTANT]
> 本项目与 Maid Storage Manager、Touhou Little Maid 及其作者、维护团队不存在隶属关系，也不代表上游项目提供官方支持或背书。扩展导致的问题请提交到[本仓库 Issues](https://github.com/hzbj/maid-storage-manager-extension/issues)，不要提交给上游项目。

## 功能模块

### 物流模块（核心玩法）

#### 1. 网络仓库

将分散在不同地点的仓库集中到一个界面中管理。玩家无需亲自往返各个仓库，就能查看库存、了解仓管和快递员的工作状态，并向指定仓库提交取物需求。请求会进入真实的女仆物流流程，由仓管备货、快递员运输，让远距离取物从“玩家跑仓库”变成“女仆把物资送过来”。

#### 2. 女仆接送

为玩家提供由司机女仆执行的远距离出行服务。玩家可以在地图上选择出发地和目的地，呼叫空闲司机前来接送；旅途中由女仆自动驾驶，抵达后安全降落，也可以在途中由玩家接管。它让已经培养的女仆不仅参与物资运输，也能成为玩家探索和往返据点时的交通力量。

#### 3. 快递物流

连接主人、仓管与不同仓库之间的物资流转。快递员会根据需求完成取货、送货、退回和异常重试，使仓库中的物资能够跨区域送达，而不再局限于仓管女仆自己的工作范围。运输过程中会持续保留物品记录，并在暂时无法交接时等待恢复，降低物流中断造成的物品丢失风险。

#### 4. 仓库维护

让仓库库存保持可查询、可整理、可调度。仓管女仆能够定期巡检工作区、更新库存清单，并将暂时无法归类的物品重新送往合适的容器。网络仓库与快递物流由此获得可信的库存信息，玩家也能减少手动查箱、整理和确认缺货的时间。

#### 5. 物流管理

集中展示仓库、物流女仆与当前任务的状态，帮助玩家判断谁在工作、物资处于哪个环节、请求是否已经完成。玩家还可以召回指定仓管、切换仓库或调整物流关系，让多名女仆参与的物流体系更容易管理和排查。

物流模块将继续围绕“物资从哪里来、由谁运输、送往哪里”扩展，逐步联动更多女仆职位；尚未实现的职位与流程不会提前写入功能承诺。

### 拓展模块（其他模组优化）

#### 1. 女仆仓管优化

补充原有仓管玩法在长期运行和复杂仓库中的维护能力，包括自动巡检、杂物回流、物品 NBT 匹配与可达性排查；同时修复已确认的显示问题，让仓管工作更稳定、更容易定位异常。

#### 2. 任务调度

允许玩家召回指定仓管，并查看巡检、请求与运输进度。它为原本各自工作的女仆补充明确的任务反馈，让玩家更容易判断任务是否开始、当前执行到哪里，以及是否遇到阻塞。

#### 3. 界面与状态增强

把本扩展的设置和工作状态整合到玩家熟悉的女仆界面与进度表中。玩家可以直接查看巡检、整理、请求和运输状态，不必只通过女仆是否移动来猜测任务是否正常执行。

#### 4. 模组联动优化

改善车万女仆、女仆仓管及《车万女仆：万法皆通》在物流场景下的协作体验，使远程交互、女仆识别、物品交接和异常恢复使用一致的物流规则，减少不同模组机制衔接时的割裂感。

拓展模块只对依赖模组进行兼容、修复和体验增强，不修改或替换上游 JAR。

## 安装

1. 从 [Releases](https://github.com/hzbj/maid-storage-manager-extension/releases) 下载普通版扩展 JAR。
2. 按上表安装必需依赖；专用服务器不需要安装 Cloth Config。
3. 将扩展 JAR 放入客户端与服务端的 `mods` 目录，确保双方使用同一版本。
4. 启动前不要删除或替换女仆仓管本体，本扩展通过独立 Mod 方式加载。

“快递员”任务无需前置道具即可启用。每次运输开始时才按当前位置选择路线并检查对应道具：

- 同时靠近主人和仓管：步行，不需要运输道具；
- 远离仓管、靠近主人：真实可用库存中需要“魔理沙的扫帚” (`touhou_little_maid:broom`)；
- 靠近仓管、远离主人：饰品栏需要装备“末影腰包” (`touhou_little_maid_spell:ender_pocket`)；
- 同时远离主人和仓管：需要扫帚与末影腰包。

缺少本次路线所需道具时只会阻止该次运输启动，不影响选择快递员任务。玩家游玩不需要安装 JDK，JDK 17 仅用于从源码构建。

## 完整功能说明

- 杂物箱：把“杂物箱标记”直接放进贴在箱子上的物品展示框。仅在“保持现状”分类模式中、过滤箱和已有同类物品箱都无法接收时作为最后兜底。
- 杂物箱回流：每轮先按女仆当前位置逐次选择最近箱子，并且只有女仆到达可交互站位、官方查看上下文完整结束、最新缓存写入成功后才算巡检完成。全部箱子成功后，一个杂物箱生成一个源批次：一次筛选其中所有可归类精确物品，按物品轮询装入女仆空槽，再在同一目标上连续投放多种物品；槽位不足时自动分批。目标顺序固定为“接受该精确物品的白名单过滤箱（即使为空）→ 最新缓存中已有同类物品的普通箱”。找不到目标时原物留在杂物箱，绝不会拿空普通箱兜底。目标空间不足时余量继续尝试匹配目标，全部满后多种余量一次退回唯一原杂物箱；暂时退不回时保留精确 NBT 账本并重试。
- NBT 匹配：每只仓管女仆都有独立的“整理时匹配 NBT”开关，旧存档默认关闭。关闭时，普通目标箱的“同类”只比较物品 ID；开启时，必须同时匹配完整 NBT。该开关不覆盖白名单箱自身的 NBT 过滤规则，并且两种模式下的实际提取、运输和回源都始终按完整 NBT 记账。
- 扩展设置：打开仓管女仆界面右侧的车万女仆“全局设置”标签，在“女仆仓管 → 女仆仓管扩展”子分类中调整当前仓管女仆的扫描半径、任务铃范围/超时/停留、刷新反馈、状态和杂物 NBT 匹配；这些项目不再与车万女仆或女仆仓管的全局分类同级。巡检周期、快递员授权和信箱审批移动到仓管任务设置末尾的“女仆仓管扩展设置”入口。修改 NBT 开关时，已提取批次先安全送达或回源，未开始计划作废并立即安排新的完整巡检。这是一次性重规划：周期巡检即使设为关闭也会执行本次，完成后仍保持关闭。
- 周期巡检：给仓管女仆装备“库存巡检器”，选择关闭、5、15、30 或 60 分钟。只扫描已加载区块，按三维球体覆盖多层仓库，每 tick 最多分析 16 个候选；逐箱查看时每走完一段都会按女仆当前位置重新选择最近的待查箱。
- 清单刷新：潜行右键装有空库存清单或已写库存清单的展示框，绑定巡检器后再把它装备给女仆。只有完整巡检结束才会发布新清单；展示框效果、主人动作栏和扩展设置页会显示发布结果。
- 可达调试：执行 `/maid_storage_manager debug_reachable` 或 `/maid_storage_manager_extension debug_reachable`，再点击仓管女仆。绿色为可达，黄色为无安全站位或路径，红色为不支持或被权限规则排除；标记仅执行者可见 15 秒。
- 终端账号恢复（OP）：执行 `/maid_storage_manager terminal_account reset_password <账号名>` 生成一次性重置码并撤销该账号全部终端令牌；执行 `/maid_storage_manager terminal_account unbind_maid <女仆 UUID>` 可强制解除异常名册绑定。两个命令也可使用 `/maid_storage_manager_extension` 根命令。
- 任务铃：手持任务铃右键自己的仓管女仆进行绑定。放置后任意玩家可按铃召回同维度、已加载且在配置范围内的绑定女仆；女仆会选择铃旁安全站位，默认 20 秒超时，到达停留 10 秒。
- 快递仓库信箱：注册名仍是 `courier_warehouse_station`，旧存档不会丢方块。未标记的信箱右键地面会记录以点击点上方为中心、平整且上空无遮挡的 3×3 起落区；再次右键其他地面才放置信箱。信箱和起落区必须同维度且相距不超过 64 格，信箱本身不能占用 3×3 净空；潜行右键可重新标记或重新发起绑定。
- 信箱绑定：放置后会搜索自身 64 格内的仓管，优先选择放置者自己拥有的最近仓管。仓管主人亲自放置时站点直接成立，不需要快递员批准；为其他玩家的仓管代放时，仓管主人可在扩展设置的“待审批信箱站点”中同意或拒绝。审批不要求仓管工作区覆盖信箱，也不预先要求完整步行路径。
- 快递员：无需装备任何运输道具即可选择“快递员”任务；开始运输时才按实际路线检查扫帚或末影腰包。每名快递员最多绑定 6 个仓管；仓管已授权的所有快递员共享其获批信箱，女仆设置页和女仆联络终端均可选择默认站点。终端不是多仓管绑定的前置条件。
- 自动运输分流：扩展只按 X/Z 水平距离判断“近”或“远”，阈值使用可配置的扫帚飞行距离；距离决定本次路线的最低道具需求和是否需要扫帚。未装备末影腰包时，靠近主人按原路线步行返回交接；只要末影腰包已装备，本次运输就保留远程交接语义，即使主人就在快递员附近也不会返回主人身边，而是在仓管交接点完成事务。远离仓管时相应使用“扫帚＋末影腰包”组合路线。
- 末影腰包远程链路：末影腰包传输段不限制主人与快递员的距离或维度。远程打开快递员设置时，站点列表、运输确认、默认站点及仓管审批都通过《车万女仆：万法皆通》的所有权校验会话解析真实女仆，不依赖客户端是否仍在追踪实体；扫帚航段仍遵守下述主世界限制。
- 扫帚约束：只有本次运输实际选择扫帚路线时才要求绑定有效的快递信箱；单纯携带扫帚不会限制步行或末影腰包路线。扫帚运输只支持主世界；快递员、仓管站点或主人不在主世界，以及任何跨维度扫帚路线都不会启动。返程时主人离开主世界，女仆会在空中悬停并提示主人，主人返回后继续。每次扫帚航段创建前都会重新校验起飞净空；信箱被破坏、审批失效或 3×3 起落区被遮挡时不会起飞，终端会显示站点无效。
- 扫帚接驳：仓管交接后，女仆先用车万女仆原生寻路走回固定 3×3 起落点，再生成扫帚垂直升空；若原生路径合法地停在标记区边缘外一格，只有女仆脚下实时复验为新的平整、露天、净空 3×3 时才允许从当前位置起飞，不会瞬移。连续寻路停滞或扫帚生成失败会提示主人并重试。抵达仓库时先落在站点、收起扫帚，再用原生寻路步行交接。返回主人时只按 X/Z 距离进入 32～64 格可配置落地区，选择平整且上空无遮挡的空地下降；落地后恢复跟随主人，或按快递员设置切换为原地 Home。扫帚路线不会调用地面安全传送。
- 独立加载：选择快递员任务或持有在途快递日志的女仆会像物流工作实体一样，以 Minecraft 实体 Tick 等级持续加载自身、扫帚、当前仓管、信箱和起落点；仓管实际处理需求、入库和退货时还会持续加载自身当前位置、完整工作范围、当前寻路目标及交互箱，因此跨区块取放物品、临时切换任务或主人离开模拟距离后仍能继续，终端也不会因实体卸载停止更新或丢失目标女仆名称。
- 附近交接：快递员进入仓管或主人 2.75 格内即执行清单或物资交接，只暂停快递员自身，不再强制双方转向、停止仓管导航或额外等待 1 秒。仓管沿用官方请求、白名单、缺货和放置判定；需求成功或失败都会返还清单，失败清单优先放在副手。交付箱行为不变。
- 满载兜底：运输容量每次读取女仆当前真实可用栏位，包含额外背包和主副手。快递员、仓管或主人满载时，超出容量的完整 NBT 物资进入持久化防丢失暂存并等待空间；当前运输方式的必要道具被取下时只暂停事务，补回后自动继续。
- 女仆联络终端：沿用旧物流查看表的注册名与绑定数据。首次使用需创建或登录服务端账号；登录令牌只写入当前终端物品，退出或密码重置可撤销。手持已登录终端右键自己的女仆注册名册，潜行右键已获批信箱注册仓库。网络仓库左栏按“信箱/仓库”展开快递员，显示仓管执勤、快递员事务与运输方式；左键信箱或地图仓库会激活对应仓库，右键则对该信箱全部在岗仓管请求立即巡检，并在终端内显示结果。一个信箱对应一个仓库，多名仓管必须绑定同一库存清单展示框；终端严格显示该展示框发布的最新清单。账号信箱与女仆只能从顶部小型“设置”按钮进入的独立设置页解绑。取物只原位更新快递员主手或副手已有的请求列表，两手无列表时拒绝。终端顶部功能页签仍只保留“网络仓库”和“女仆接送”。
- 女仆接送：在终端地图短按选择起点和终点，拖动超过阈值只平移地图；起点默认下单玩家当前位置，终点可以位于客户端未知区域。主世界内的空闲司机携带扫帚前往起点，玩家上车后以“女仆第一乘客”标识由扩展服务端自动驾驶，客户端原版扫帚控制同步停用；载客航线采用更高巡航净空并按前方地形抬升。正常抵达会安全落地；终点 12 格内无安全落点或玩家主动结束行程时，玩家改为第一乘客并恢复原版驾驶；下车后女仆重新接管并安全降落。
- 界面画风：女仆联络终端采用轻装饰的深色分栏与高对比文字；地图读取客户端已经加载的原版实际地形，不会为了制图额外加载远方区块。地图圆点颜色固定为自己绿色、女仆黄色、当前可见的其他玩家白色；仓库用可点击的房屋图标表示。只有女仆接送页显示带“起/终”标记和图例的路线，不绘制含义不明的关系线。
- 进度表状态：绑定原仓管“进度表”后，左上角非任务格区域显示状态图标，女仆名下方显示简短状态文字。可区分巡检及其发现/检查/清理/清单刷新阶段、任务铃寻路/等待、请求、物流、存放、查看、整理、分类、进食、交流、协作、拾取、普通寻路、战斗、休息、睡觉和待机；巡检发现、逐箱检查与杂物箱清理都会显示完成数/总数。清理进度按唯一源杂物箱计算，同一箱分批不重复计数，余量退回源箱前不提前完成，保存重载后进度仍保留。小展示框自动使用紧凑文字，任务格数量保持原样。纸面文字统一为黑色，物品图标上的堆叠数量保留白色。

## 问题反馈

提交问题前请确认客户端与服务端使用相同版本，并附上：

- 本扩展、Forge、车万女仆、女仆仓管和魔法皆通的完整版本；
- 单人游戏或服务端环境，以及能否稳定复现；
- `logs/latest.log` 和崩溃报告；
- 复现步骤、预期结果和实际结果。

扩展关闭后不再出现的问题，请提交到[本仓库 Issues](https://github.com/hzbj/maid-storage-manager-extension/issues)。无法确认来源时也请先在本仓库排查，不要直接打扰上游维护者。

## 构建

使用 JDK 17；构建始终输出兼容 Java 17 的字节码。女仆仓管和车万女仆从 Modrinth Maven 的精确版本 ID 获取，Tour Guide 1.0.2 暂作为本地编译依赖保存在 `libs/`：

