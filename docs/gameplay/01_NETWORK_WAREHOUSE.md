# 01 网络仓库策划案

> 状态：终端账号、信箱/仓库树、库存查询、取物下单与存物入口已实现；物理快递箱、置物架和信箱存放区仍待后续站点基础设施阶段
> 实施优先级：第一项
> 共用规则见 [女仆物流玩法总纲](../GAMEPLAY_DESIGN_MAID_LOGISTICS.md)。

## 1. 功能目标

网络仓库直接服务玩家：玩家在女仆联络终端中查看已获批仓库的库存，发布取物或存物订单，由仓管实际备货、快递员实际运输。

网络仓库不是跨维度无限背包，也不实现第二套库存系统。终端只把现有“库存清单 → 需求列表 → 仓管处理 → 末影腰包/快递员交接”流程整合成一个 UI 和自动化接口。

### 1.1 固定底层契约

- 查询来源只能是仓管发布的库存清单。
- 玩家确认取物后，终端原位更新快递员主手或副手已有的标准女仆仓管需求列表，写入所选物品、数量和目标仓管；保留已有任务 UUID 和其他自定义标签。
- 仓管继续按原需求列表行为找货、报告缺货、返还列表；终端不直接从箱子抽取物品。
- 需求列表作为真实 `ItemStack` 参与现有快递事务，不用一份新的网络订单数据替代它。
- 终端从需求列表标签读取请求、已收集、缺少、失败和完成状态，并转换成 UI 进度。
- 玩家与远处快递员之间的远程链路继续由末影腰包承担；没有末影腰包时不得伪造远程交接。
- 快递箱只承载已经由需求列表实际取得或准备存入仓库的货物。

## 2. 参与对象

- 玩家与女仆联络终端；
- 当前终端账号名册中选中的一名快递员；
- 一个或多个绑定仓管女仆的快递信箱；
- 仓管已发布的库存清单；
- 快递员主手或副手已有的一张可编辑女仆仓管需求列表；
- 快递员装备的末影腰包，用于已有远程交接段；
- 信箱的快递存放区与快递置物架；
- 玩家本人、交付箱或另一个获批信箱作为送达目标。

首版继续复用当前“每名快递员最多绑定 6 个仓管站点”的限制，不在同一阶段重写全部授权结构。

## 3. 终端中的网络仓库 UI

### 3.1 地图区域

- 使用客户端已加载区块生成实际原版地形底图，不额外加载或生成远方区块；未加载区域明确显示为无数据。
- 实体使用圆点显示：玩家自己为绿色、仓管与快递员女仆为黄色、客户端当前可见的其他玩家为白色，悬停显示名称和坐标。
- 网络仓库页不显示无说明的黄色关系线；站点绑定关系由左侧仓库节点列表表达。
- 支持拖拽平移、以鼠标所在位置为中心滚轮缩放，以及一键回到玩家。
- 信箱图标显示在线、库存清单新鲜度、存放区剩余箱位、起落区有效性和当前等待快递数。
- 点击信箱可将库存列表限制为该仓库；选择“全部仓库”时显示聚合结果。

### 3.2 左侧信箱与快递员

- 第一层只列出已注册到当前终端账号的信箱及其对应仓库，并显示信箱有效性、仓管是否加载、是否正在使用仓管职业执勤。
- 点击信箱展开账号名册中的快递员；每行显示在线/空闲/工作中、真实事务阶段和当前步行、末影腰包、扫帚或组合运输方式。
- 点击快递员会选中她，并通过服务端权限校验把当前信箱绑定为其可用仓库和默认站点；左键信箱或地图仓库图标也会激活该仓库。顶部账号控件旁的小型“设置”按钮会进入独立设置页，信箱/仓库从账号注销和快递员名册解绑只在该页提供。右键信箱行或地图仓库图标会先激活对应仓库、再请求立即巡检，并在当前终端页显示服务端返回的成功或失败原因。
- 展开列表只是账号调度入口，不绕过原有仓管主人审批、每名快递员最多 6 个仓管和事务互斥规则。

### 3.3 库存面板

- 物品图标、名称、可用数量、所在仓库数；
- 搜索、排序和仅显示可取物品；
- 展开物品后显示各仓库数量与最近清单时间；
- 数量选择、来源策略和“加入订单”；
- 库存清单超过 **7 个游戏日（168000 tick）** 时标记陈旧；UI 明确显示“需要重新巡检”，不把缓存冒充实时库存。
- 仓管完成刷新时会把新清单 UUID 写入巡检器所绑定的展示框，同时发布到仓管的持久网络引用；终端查询优先读取展示框现物，卸载时回退到最后已发布引用。

### 3.4 订单面板

- 取出或存入；
- 物品与数量；
- 指定仓库或自动选择；
- 送达玩家、交付箱或目标信箱；
- 预计需要的快递箱数量；
- 预计运输方式与当前阻塞原因；
- 确认、取消和查看进度。

## 4. 远程取物流程

1. 玩家搜索物品并选择数量。
2. 系统根据库存清单给出能完整满足的仓库；首版一张订单只选择一个仓库，不跨多个仓库拆单。
3. 玩家选择送达目标并确认。
4. 服务端重新检查权限、仓库状态、快递员状态、末影腰包条件和存放区容量。
5. 终端原位更新快递员手持的标准需求列表；若两手都没有可编辑清单，流程在此拒绝，不创建物品。
6. 仓管按原有需求列表、过滤和缺货规则实际取出物品并返还列表。
7. 已取得货物按每箱最多 14 格、单一目标规则打包。
8. 快递箱出现在仓管信箱的存放区或置物架。
9. 快递员接单并运输快递箱。
10. 送达玩家时执行附近交接；送达交付箱时安全入箱；送达另一信箱时生成目标区快递箱。
11. 终端读取返回需求列表并显示实际交付、缺货和失败结果，不凭空补齐。

末影腰包只承担现有远程交接段，不能绕过库存清单、手持需求列表、仓管取货和快递事务账本。终端不会向玩家或女仆背包创建清单草稿。

## 5. 远程存物流程

1. 玩家选择目标仓库，或允许系统按过滤规则与容量推荐仓库。
2. 玩家选择要存入的自身物品，创建“上门取件”订单。
3. 快递员前往玩家附近完成交接，把物品写入一个或多个目标明确的快递事务。
4. 快递员把货物送到仓管信箱存放区，生成快递箱。
5. 仓管接收快递箱，按现有白名单、已有同类物品和杂物箱兜底规则分类。
6. 成功入库的物品提交订单；无处存放的原物退回快递箱并等待换仓或退回玩家。

首版不允许终端直接从玩家背包无动画移走物品。必须由快递员到场交接，或由现有末影腰包远程链路完成明确的交接段；仓管入库仍走现有分类流程。

## 6. 库存聚合规则

- 仓库库存来源继续使用仓管发布的书面库存清单和完整巡检结果。
- 相同物品默认按物品 ID 聚合；开启精确 NBT 查看时按完整 NBT 分行。
- UI 显示的是“清单可用量”，下单执行时必须再次由仓管确认真实库存。
- 同一个物品存在于多个仓库时，自动选择顺序为：能完整满足 → 清单较新 → 路线可用 → 距离较近。
- 首版不建立复杂的跨订单库存预留中心；同一时刻对同一仓管只允许一个会修改库存的快递/整理事务，避免超卖。

## 7. 快递箱规则

- 一张取物或存物订单可以拆成多个快递箱。
- 单箱 14 格；同一物品可使用正常堆叠上限。
- 一个箱只能指向一个玩家交付目标、交付箱或信箱。
- 送达目标可在下单时选择玩家当前位置或已设置的固定交付箱；两者都受当前维度、权限和路线校验。
- 玩家取消尚未取货的订单时直接释放任务；已经装箱后必须完成退回仓库，不能直接删除箱子。
- 仓管、快递员、玩家或目标容器满载时，货物继续保存在快递箱/事务账本中。
- 玩家可用 Carry On 搬起并重新放置快递箱；搬运时订单暂停自动处理。
- 快递所有者或获准玩家右键拆包时，箱内物品变为一地世界拾取物，订单结束为“已人工拆包”。

## 8. 失败与恢复

| 情况 | 处理 |
| --- | --- |
| 库存清单陈旧 | 允许玩家查看，但确认订单前提示并要求重新巡检或接受可能缺货 |
| 仓库缺货 | 交付实际数量，缺少部分标记失败 |
| 来源存放区已满 | 仓管停止装新箱，订单等待空位 |
| 没有可用快递员 | 快递箱保留在来源站点排队 |
| 目标失效或已满 | 快递员携箱等待、改选获批目标或退回来源 |
| 快递员中途失去道具 | 保留事务并暂停，补回后继续 |
| 服务器重启 | 从订单、快递箱和 `CourierData` 共同恢复唯一事务 |
| 取消在途订单 | 快递员把原物退回来源信箱，完成后才结束订单 |
| 玩家用 Carry On 搬箱 | 标记玩家搬运中；重新放入有效存放区后恢复，放在网络外则保持人工保管 |
| 玩家右键拆箱 | 生成完整 NBT 拾取物，删除箱体并把订单标记为已人工拆包 |

## 9. 与现有代码的复用

- `logistics_tracker`：迁移为女仆联络终端；
- `LogisticsTrackerScreen/Service/Snapshot`：升级为公共 UI、服务端快照和仓库页面；
- `CourierData`：继续保存仓管绑定、运输方式和在途账本；
- `CourierService`：继续执行步行、扫帚、末影腰包、取物和存物事务；
- `InventoryListRefreshService`：提供仓管最新库存清单；
- 女仆仓管 `RequestListItem`：作为终端自动下单后的真实需求载体和进度来源；
- `courier_warehouse_station`：升级为三合一快递信箱；
- 现有交付箱、附近交接、满载暂存、召回和清除工作继续保留。

## 10. 首版不包含

- 跨多个仓库自动拆单；
- 玩家之间共享仓库网络；
- 自动循环补货；
- 餐厅菜单与烹饪；
- 砍树、挖矿等职业自动打包；
- 女仆接送的司机与乘客事务（已在独立页面实现）；
- 完整的全世界地形地图记录。

这些内容分别归入后续功能或迭代，不阻塞网络仓库首版。

## 11. 首版验收标准

- 玩家手持女仆联络终端右键进入网络仓库页面。
- 可以查看至少一个已绑定仓管的清单、搜索物品和看到更新时间。
- 快递员手持可编辑需求清单时可以创建一次取物订单；终端原位更新该清单，由仓管实际备货、生成快递箱、快递员运输并交付。
- 可以创建一次存物订单，由快递员取件、仓管实际入库。
- 送达目标可选择玩家当前位置或固定交付箱。
- 库存清单在 7 个游戏日后正确标记陈旧。
- 快递箱可被 Carry On 搬运，也可由获准玩家右键拆成世界拾取物。
- 快递箱、玩家、女仆或目标满载时不丢物、不复制。
- 旧物流查看表绑定、仓管站点和在途快递能够迁移并继续运行。

## 12. 已确认参数

1. 单个快递箱为 **14 格**。
2. 首版一张取物订单只选择一个仓库，多仓拆单留到后续。
3. 玩家取物可选择送到玩家当前位置或固定交付箱。
4. 库存清单默认超过 **7 个游戏日（168000 tick）**后标记陈旧。
5. 快递置物架单列最高 **4 层**。
6. 网络仓库只自动化末影腰包、库存清单和需求列表的既有处理链。
7. 快递箱支持 Carry On 搬运，并允许获准玩家右键拆成世界拾取物。
