# 02 餐厅外卖策划案

> 状态：仅拆分，等待网络仓库完成后再确认
> 共用规则见 [女仆物流玩法总纲](../GAMEPLAY_DESIGN_MAID_LOGISTICS.md)。

## 1. 功能目标

餐厅外卖直接服务玩家：玩家在女仆联络终端中选择餐厅、菜品、数量和送达目标，厨师实际取得原料并制作，快递员把成品送达。

## 2. 餐厅站点

- 使用通用快递信箱，不新增餐厅专用方块。
- 信箱绑定一名厨师或餐厅职业女仆。
- 信箱的快递存放区同时承担“食材收货区”和“成品取餐区”。
- 同一个快递箱只处于食材单或成品单的一种阶段，避免厨师和快递员同时抢占。

## 3. 玩家下单

1. 终端切换到“餐厅外卖”。
2. 地图显示已获批餐厅信箱、营业状态和可送达范围。
3. 玩家选择餐厅、菜品、数量和送达目标。
4. 系统检查成品库存、食材需求、厨师状态和快递员运力。
5. 玩家确认后建立外卖订单。

## 4. 食材补给

- 餐厅适配器声明菜品需要的原料和数量。
- 餐厅已有原料时直接进入制作。
- 缺少原料时，可以从网络仓库生成食材快递箱，目标指向厨师的快递信箱。
- 厨师从目标存放区接收食材快递，只取本订单允许的原料。
- 未使用原料保留在餐厅库存或生成退回仓库的快递，具体策略由玩家配置。

## 5. 成品配送

- 厨师完成菜品后使用同一种通用快递箱打包，不新增餐盒道具。
- 成品快递记录餐厅、顾客、送达位置和订单 ID。
- 快递员从餐厅信箱取走成品，送给玩家、交付箱或另一个信箱。
- 玩家离线时按订单设置等待、放入交付箱或退回餐厅。

## 6. 兼容层级

- 深度适配：读取菜单、原料、制作进度并安全提交食材。
- 协调适配：玩家配置菜单与输入/输出规则，餐厅附属自己完成烹饪。
- 只读兼容：显示厨师和位置，但必须在原附属界面操作。

没有适配器时不直接向厨师背包塞原料。可以先把食材快递送到其信箱，等待玩家或附属自己的输入机制处理。

## 7. 首版边界

- 只选择一个明确支持的餐厅附属做首个深度适配。
- 不做服务费、评价、保温或食物品质系统。
- 不新增餐盒、菜单板或餐厅铃。
- 网络仓库只是可选补货来源；餐厅已有食材时可独立接单。

## 8. 待后续确认

1. 首个适配的餐厅模组及版本。
2. 菜单由附属自动读取，还是允许玩家手动发布成品目录。
3. 未使用食材默认留在餐厅还是退回仓库。
4. 玩家离线时外卖默认等待还是送到固定交付箱。
