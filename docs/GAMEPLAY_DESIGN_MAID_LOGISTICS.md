# 女仆物流玩法总纲

> 状态：公共规则已确认；终端账号、网络仓库接口与女仆接送首版已实现，物理快递箱/置物架仍待后续站点基础设施阶段
> 当前代码基线：`v1.3.2`
> 本文只定义四项功能共用的规则；每项功能在独立策划中单独确认、单独实现。

## 1. 模组定位

本模组的长期定位是**连接不同职业女仆的联络与物流扩展**。

女仆本体及各职业附属继续负责砍树、挖矿、烹饪、仓储等具体工作。本模组提供统一地图终端、快递站点、货物打包、运输调度和职业间交接，不把所有职业改造成一套通用 AI。

## 2. 四项功能独立拆分

| 功能 | 核心服务对象 | 主要内容 | 独立策划 |
| --- | --- | --- | --- |
| 网络仓库 | 玩家 | 远程查库存、下单取物、派送存物 | [01 网络仓库](gameplay/01_NETWORK_WAREHOUSE.md) |
| 餐厅外卖 | 玩家 | 点餐、食材补给、厨师制作、成品配送 | [02 餐厅外卖](gameplay/02_RESTAURANT_DELIVERY.md) |
| 女仆接送 | 玩家 | 地图选起终点、呼叫接送女仆、实际乘坐 | [03 女仆接送](gameplay/03_MAID_TRANSPORT.md) |
| 派遣与职业编排 | 女仆之间 | 设置职业、工作区、时间和货物流向 | [04 派遣与职业编排](gameplay/04_DISPATCH_COORDINATION.md) |

四项功能共用女仆联络终端、快递信箱、快递箱、快递置物架、权限、地图和快递员队列，但不再混写成一份大功能。

当前已落实**女仆联络终端账号、网络仓库与女仆接送**；餐厅外卖和派遣编排不显示入口，也不提前实现。

## 3. 女仆联络终端

### 3.1 名称与交互

- 正式名称：**女仆联络终端**。
- 沿用现有 `maid_storage_manager_extension:logistics_tracker` 注册 ID，旧物品和绑定数据不作废。
- 玩家必须手持终端并右键，才打开全屏地图界面。
- 不制作第一人称手持地图渲染、常驻小地图或正常游玩画面中的持续状态显示。
- 未登录时先创建或登录服务端账号；终端物品只保存账号 ID 和可撤销设备令牌，不保存密码及密码哈希。
- 同一账号可登录任意终端；一名快递员只能属于一个账号。持有已登录终端等于持有该账号的调度权限。
- 手持已登录终端右键自己的快递员女仆注册名册，潜行右键已获批信箱注册仓库；旧物流查看表绑定的快递员在首次登录后迁移为默认网络仓库快递员。
- 网络仓库和女仆接送分别选择默认快递员与司机，同一女仆可兼任，但事务互斥。
- 密码使用 PBKDF2-HMAC-SHA256 加盐保存。OP 一次性重置码会撤销全部终端令牌，使用后立即失效，并强制在终端设置新密码。

叙事只保留一句：终端会记录获准联络的女仆、快递站点和货物流向。无需再为它建立复杂的河童机关或天狗通信背景。

### 3.2 共用 UI 外壳

```text
┌──────────────────────────────────────────────────────────┐
│ 账号 / 退出       [网络仓库] [女仆接送] [设置]          │
├──────────┬───────────────────────────────┬───────────────┤
│ 女仆与   │                               │ 当前功能面板  │
│ 站点列表 │         可拖拽缩放地图         │ 库存/菜单/    │
│ 图层筛选 │  实际地形、实体圆点与区域叠加   │ 路线/工作设置 │
├──────────┴───────────────────────────────┴───────────────┤
│ 当前订单、快递队列、阻塞原因和快捷处理                    │
└──────────────────────────────────────────────────────────┘
```

地图交互参考 Xaero 世界地图的成熟习惯：以实际地形作为主体，左键拖动、滚轮缩放、列表与地图悬停联动。只参考交互方式，不复制其代码、资源或地图数据，也不把 Xaero 设为必需依赖。地图实体统一使用圆点：自己绿色、女仆黄色、当前可见的其他玩家白色。网络仓库页不默认绘制关系连线；以后若派送编排需要线路，必须作为带方向、图例和显隐开关的独立图层。

当前功能页签只显示“网络仓库”和“女仆接送”。账号控件旁另有一个小型“设置”按钮，进入独立页面管理信箱/仓库和快递员名册解绑；它不是第三个玩法页签。餐厅外卖、派遣与职业编排在正式开发前不占用顶部入口；以后仍复用同一件终端道具。

### 3.3 网络仓库的接口边界

女仆联络终端中的网络仓库只提供 UI 与自动化操作入口，不另造一套虚拟仓库、取货 AI 或物品传输规则：

- **库存展示**读取仓管已经发布的库存清单；
- **取物下单**由终端原位更新快递员主手或副手已有的女仆仓管需求列表；没有手持清单时无法取物，终端不会生成新清单；
- **远程交接**继续通过快递员装备的末影腰包链路处理；
- **仓管执行**仍由女仆仓管根据需求列表完成找货、缺货、过滤和返还；
- **状态显示**把库存清单时间、需求列表进度和快递事务转换成玩家易读的 UI。

没有末影腰包时，终端仍可查看库存清单和准备需求列表，但不能绕过现有距离与运输条件完成远程交接。

## 4. 快递信箱：统一女仆物流站点

沿用现有 `courier_warehouse_station` 注册 ID，显示与用途从“快递仓库信箱”扩展为通用的**快递信箱**。旧存档中的仓库信箱继续有效。

每个信箱整合三个配置：

### 4.1 快递员起落区

- 沿用现有露天、平整、无遮挡的 3×3 起落区规则。
- 信箱和起落区位于同一维度，并保持现有距离与安全校验。
- 使用扫帚的快递员在这里起飞、降落、召回和等待。
- 起落区不能与快递存放区重叠，避免快递箱阻挡飞行。

### 4.2 快递存放区

- 玩家在女仆联络终端中选中信箱，再在游戏中框选快递存放区（辅助UI会在放置信箱后消失）。
- 存放区由一组可放置快递箱的方块格组成；必须与信箱同维度并位于允许范围内。
- 没有置物架的格子只能在地面放一个快递箱。
- 放有快递置物架的格子可以沿 Y 轴向上堆叠置物架，增加同一水平位置的快递箱容量。
- 存放区满时，生产女仆停止工作并停止生成新快递并显示“等待空位”，不能把货物丢在地上。

### 4.3 绑定女仆

- 每个信箱可绑定多名同种工作状态的女仆，代表该工作区域的快递收发站。
- 右键放置好的信箱可以更换绑定的女仆（女仆和信箱需要为同一主人）。
- 绑定需要女仆所有者批准，并记录稳定 UUID；改名不破坏绑定。
- 信箱不直接改变女仆职业。职业识别与工作设置仍由对应附属适配器处理。
- 快递员不独占某个信箱；获准的快递员可以服务多个信箱和多条连接。

### 4.4 信箱有效状态

信箱只有同时满足以下条件才可参与自动物流：

- 已绑定并获批女仆；
- 起落区配置有效，或当前运输方式不需要扫帚；
- 快递存放区至少有一个有效位置；
- 信箱、存放区和绑定女仆的维度/距离满足该职业适配器要求；
- 玩家对该信箱和女仆有访问权限。

## 5. 快递箱

### 5.1 定义

- 快递箱在世界中占用 1×1×1 方块空间。
- 容量确定为 **14 格**，即原版单箱 27 格的一半向上取整。
- 快递箱不是新增的可合成、可随意携带道具，而是由物流系统生成和管理的任务容器。
- 快递箱是带方块实体的真实方块，可以被兼容的搬运模组整体移动，但不能作为物品直接放入普通背包。
- 玩家可以查看属于自己的快递箱及目标，也可以主动拆包，但不能把它当成廉价普通箱长期使用。
- 箱内物品始终保存完整 NBT。

### 5.2 必要数据

每个快递箱至少记录：

- 唯一快递 ID；
- 来源信箱和来源女仆；
- 目标信箱、目标女仆或玩家交付目标；
- 所属功能：网络仓库、外卖或派送；
- 箱内精确物品清单；
- 创建者、所有者和授权玩家；
- 当前状态与在途快递员；
- 创建时间、重试次数和失败原因。

### 5.3 生命周期

`等待打包 → 已放入来源存放区 → 等待快递员 → 运输中 → 已放入目标存放区 → 目标女仆接收 → 完成`

异常状态包括：

- 来源存放区已满；
- 没有可用快递员；
- 目标信箱无效或已满；
- 目标女仆忙碌、离线或不再兼容；
- 已安全暂停；
- 正在退回来源；
- 玩家搬运中；
- 已人工拆包；
- 等待玩家处理。

### 5.4 目标女仆接收

- 仓管女仆接到快递后，把箱内物品交给现有分类与仓储流程。
- 厨师接到食材快递后，由餐厅适配器把允许的原料交给烹饪流程。
- 其他职业只能接收其适配器明确声明可处理的物品。
- 接收失败时快递箱保留原物和目标，不自动散落或改派给另一名女仆；只有获准玩家主动右键拆包才会散落物品。

### 5.5 玩家搬运与拆包

**Carry On 兼容**

- 快递箱设计为标准方块实体，使 [Carry On](https://www.curseforge.com/minecraft/mc-mods/carry-on) 能像搬运有内容的箱子一样保留方块状态、元数据、库存和 NBT。
- Carry On 是可选兼容，不作为本模组必需依赖；未安装时不影响快递自动化。
- 玩家搬起快递箱时，订单进入“玩家搬运中”，原存放位立即释放，快递员和目标女仆不再争抢该箱。
- 玩家重新放下后，以方块实体中的唯一快递 ID 恢复订单位置。
- 放在目标信箱存放区内时，目标女仆可以继续接收；放在网络外时视为人工保管，不再自动派送。
- 服务端只承认一个快递 ID 对应的一个有效方块/在途事务；出现重复 ID 时冻结重复对象，避免复制货物。

**右键拆包**

- 快递所有者或获准玩家可以直接右键快递箱执行人工拆包。
- 箱内物品按原堆叠和完整 NBT 生成世界拾取物，散落在快递箱周围；快递箱随后消失。
- 拆包成功后订单标记为“已人工拆包”，取消后续快递和目标女仆接收。
- 未授权玩家不能拆包；服务端必须先成功生成全部拾取物，再删除快递箱和提交订单状态。
- 人工拆包是明确的玩家选择，因此不走自动退回来源流程。

## 6. 快递置物架

快递置物架是本轮规划中**唯一新增的可合成、可放置道具**。

- 只能在信箱的快递存放区内参与自动物流。
- 可以沿 Y 轴向上堆叠，每一层提供一个快递箱位置。
- 快递箱在置物架层内显示为实体箱体，但货物权威数据由服务端快递记录与置物架方块实体共同校验。
- 破坏空置物架正常掉落；破坏装有快递的置物架时，快递先转入信箱的防丢失暂存，不能连箱带货直接掉落或复制。
- 单列最高 **4 层**，超过高度的置物架不计入该信箱自动快递容量。

## 7. 通用快递流程

```text
来源女仆完成产物或备货
  ↓ 按目标与容量拆分快递
来源信箱存放区生成快递箱
  ↓ 创建运输委托
共享快递员队列选择下一箱
  ↓ 实际步行 / 扫帚 / 末影腰包运输
目标信箱存放区接收快递箱
  ↓ 目标职业适配器确认接收
仓管入库 / 厨师取料 / 其他职业继续加工
```

一个快递箱只有一个明确目标。超过 14 格或需要发给不同目标的货物必须拆成多个快递箱。

一名快递员可以负责多个信箱和多个连接，但同一时间只能携带其容量和运输实现允许的快递，不能让两个事务覆盖同一背包或乘骑状态。

## 8. 兼容性底线

- 核心不直接写女仆砍树、挖矿、餐厅等附属的私有任务数据。
- 每个附属通过独立适配器声明：职业识别、工作范围、排班、可打包产物、可接收原料和安全中断边界。
- 适配分为深度适配、协调适配和只读兼容。
- 没有深度适配时，玩家仍可使用该附属原界面设置工作，并通过指定输出/输入信箱参加物流。
- 附属缺失或版本不兼容时只暂停对应配置，不删除未知任务数据，不自动切换职业。
- 快递员不能默认抽走职业女仆整个背包；只有适配器确认的产物或玩家指定交接库存可以打包。
